## 1. Vision & Positioning

### Category

**AI Instruction Quality** — a category that doesn't exist yet. Schliff defines it.

### Tagline

> **The quality score for AI instructions.**

### Narrative

```
Codecov made test coverage visible.
SonarQube made code quality a gate.
Schliff makes AI instruction quality measurable.

Zero dependencies. No LLM. Same input, same score.
```

### Why Now

- 2026 framed as "Year of AI Quality" (CodeRabbit, IBM, CIO Magazine)
- ETH Zurich study (Feb 2026, 138 tasks): poorly written AGENTS.md files **reduce** agent success by 3% and increase costs by 20%
- 60,000+ repos have AGENTS.md, SkillsMP indexes 66,500+ skills — none have quality gates
- MCP ecosystem: 10,000+ servers, 97M monthly SDK downloads — tool description quality is a growing pain
- No tool combines deterministic scoring + LLM evolution + deterministic verification

### The Unique Insight

Most tools in the AI agent ecosystem use LLMs to evaluate LLM artifacts — circular. Schliff breaks the circle: **rule-based static analysis on artifacts written for LLMs.** The same logic as ESLint for JavaScript or ruff for Python, except the "language" is instruction files.

The evolution engine adds a second unique layer: deterministic diagnosis → LLM treatment → deterministic verification. No other system has this triple determinism layer.

### Name

"Schliff" means "polish/refinement" in German ("den letzten Schliff geben" = add the finishing touches). Short, unique, no namespace conflict on PyPI/npm/GitHub. The non-obviousness is a feature — the explanation itself is marketing.

## 3. Scope Expansion: New Artifact Types

### 3.1 System Prompts (Priority 1 — zero competition, huge market)

**Full spec:** `docs/specs/system-prompt-scoring-spec.md` (written by subagent, 7 dimensions with concrete regex patterns, rubrics, and anti-gaming rules).

**7 Dimensions:**

| Dimension | Weight | What it measures |
|-----------|--------|-----------------|
| structure_prompt | 15% | Role definition, task description, constraints, output format, examples, separators, logical ordering, length, progressive detail, no dead content |
| output_contract | 15% | Format spec, length constraints, tone/voice, schema definition, required fields, forbidden content, response structure, error response format, validation instruction, example output |
| efficiency | 15% | Signal-to-noise, hedging, filler, tautologies, "CRITICAL!" begging, redundancy, caching-aware ordering |
| clarity | 15% | Contradictions (incl. system-prompt-specific like "be concise" + "explain thoroughly"), vague instructions, ambiguous scope |
| security | 15% | Prompt injection defense, identity anchoring, input sanitization, output filtering, scope limitation, canary/tripwire patterns |
| composability | 10% | Scope boundaries, handoff patterns, tool awareness, context window management, statelessness |
| completeness | 15% | Error handling, edge cases, ambiguity handling, off-topic, multi-turn, language/locale, empty input, rate limits, graceful degradation |

**Detection:** Files that are NOT markdown with frontmatter or headers, contain system-prompt-specific patterns (role definitions, output contracts), or are explicitly marked with `--format system_prompt`.

**Effort:** ~5 days (5 of 7 dimensions transfer from existing scoring).

### 3.2 MCP Tool Descriptions (Priority 2 — fast-growing market)

**6 Dimensions:**

| Dimension | Weight | What it measures |
|-----------|--------|-----------------|
| schema_quality | 25% | All parameters described, have examples, required fields correct, types specified, enum values documented |
| trigger_alignment | 20% | Would an agent choose this tool for the right use case? Name/description precision, disambiguation from similar tools |
| efficiency | 15% | Description conciseness, no redundancy with schema |
| clarity | 15% | Ambiguity in descriptions, parameter naming consistency |
| security | 15% | Overpermission patterns, input validation hints, output sanitization |
| composability | 10% | Namespace isolation, conflict check across tool set, idempotency hints |

**Detection:** JSON files with MCP tool structure (`name` + `description` + `inputSchema`), or `--format mcp_tool`.

**Effort:** ~3 days (shorter texts, more structured format).

### 3.3 Prompt Templates (Priority 3 — later)

Deferred to v8.1. Depends on stabilizing the registry pattern first.

## 4. Evolution Engine: `schliff evolve`

### Core Concept

A closed loop that no other tool provides:

```
┌─────────────────────────────────────────────────┐
│  1. DETERMINISTIC SCORE (existing engine)         │
│     Per-dimension scores + specific issues         │
│     ~0 tokens, ~instant                            │
└────────────────────┬────────────────────────────┘
                     ▼
┌─────────────────────────────────────────────────┐
│  2. DETERMINISTIC PATCHES (60-70% of fixes)       │
│     Frontmatter inserts, noise removal, dedup      │
│     ~0 tokens, ~instant                            │
└────────────────────┬────────────────────────────┘
                     ▼
┌─────────────────────────────────────────────────┐
│  3. LLM IMPROVE (via LiteLLM → any provider)     │
│     Directed gradients per dimension               │
│     Temperature 0 → higher on stagnation           │
│     Budget-tracked                                 │
└────────────────────┬────────────────────────────┘
                     ▼
┌─────────────────────────────────────────────────┐
│  4. DETERMINISTIC VERIFY + DIMENSION GUARD        │
│     Re-score all dimensions                        │
│     >threshold drop in ANY dimension → revert      │
│     Anti-gaming on evolved output                  │
└────────────────────┬────────────────────────────┘
                     ▼
┌─────────────────────────────────────────────────┐
│  5. CONVERGENCE                                    │
│     EMA plateau detection                          │
│     Strategy switch on stagnation                  │
│     Budget-aware stopping                          │
└─────────────────────────────────────────────────┘
```

### CLI Interface

```bash
# Standard: improve to grade A, max 10 generations
schliff evolve SKILL.md

# Aggressive: S-grade with large budget
schliff evolve SKILL.md --target S --generations 20 --budget 100000

# Local + free via Ollama
schliff evolve SKILL.md --provider ollama --model ollama/llama3

# Deterministic-only (no LLM, $0)
schliff evolve SKILL.md --budget 0

# Target a specific dimension
schliff evolve SKILL.md --strategy dimension --dimension triggers

# Whole repo
schliff evolve --repo .

# CI-friendly JSON output
schliff evolve SKILL.md --target 85.0 --json
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `<path>` | positional | required | Path to instruction file |
| `--target` | str | `"A"` | Target grade (S/A/B/C) or score as float |
| `--generations` | int | `10` | Max generations |
| `--budget` | int | `50000` | Token budget (0 = deterministic only) |
| `--provider` | str | auto | LiteLLM provider prefix |
| `--model` | str | auto | Specific model string |
| `--min-improvement` | float | `0.5` | Min score improvement per gen to continue |
| `--strategy` | str | `"gradient"` | Strategy: gradient, holistic, dimension |
| `--dimension` | str | None | With --strategy dimension: which one |
| `--repo` | flag | False | Evolve all instruction files in repo |
| `--json` | flag | False | JSON output |
| `--dry-run` | flag | False | Show plan without changes |
| `--lineage` | str | `~/.schliff/lineage/` | Lineage directory |

### Algorithm

```
Phase 0: INIT
  ├─ Score file (build_scores + compute_composite)
  ├─ Create lineage session
  └─ Resolve target (grade → threshold: S=95, A=85, B=75, C=65)

Phase 1: DETERMINISTIC PATCHES (free, no LLM)
  ├─ text_gradient.compute_gradients() → ranked fix list
  ├─ text_gradient.generate_patches() → concrete patches
  ├─ Apply patches one-by-one, re-score after each
  ├─ Dimension Guard check (threshold=2.0 per dimension)
  └─ Rollback individual patch if any dimension drops

Phase 2: LLM-GUIDED IMPROVEMENT (iterative, budgeted)
  ├─ Compute fresh gradients from current best
  ├─ Build LLM prompt with top-5 issues + dimension scores
  ├─ LLM generates improved version via LiteLLM
  ├─ Deterministic re-score
  ├─ Dimension Guard (threshold=3.0)
  ├─ Accept if composite improved, reject otherwise
  ├─ Plateau detection (EMA, window=5)
  └─ Escape: strategy switch → temperature bump → stop

Phase 3: FINALIZE
  ├─ Write best generation to file
  ├─ Close lineage record
  └─ Print summary
```

### Dimension Guard

```python
def dimension_guard(old_dims, new_dims, threshold=3.0):
    """No dimension may drop more than threshold points.
    Prevents improvement in one area at the cost of another."""
    violations = []
    for dim, old_score in old_dims.items():
        if old_score < 0:
            continue  # not measured
        new_score = new_dims.get(dim, -1)
        drop = old_score - new_score
        if drop > threshold:
            violations.append(f"{dim}: {old_score:.1f} → {new_score:.1f}")
    return GuardResult(passed=len(violations) == 0, violations=violations)
```

Thresholds by context:
- Deterministic patches: 2.0 (conservative, isolated patches)
- LLM generations — gradient strategy: 3.0
- LLM generations — holistic strategy: 5.0 (allows larger rewrites)

### Plateau Detection

```python
class PlateauDetector:
    """EMA-based convergence detection."""
    def __init__(self, window=5, min_improvement=0.5, alpha=0.3):
        self.window = window
        self.min_improvement = min_improvement
        self.alpha = alpha
        self.ema_delta = 0.0
        self.stale_count = 0

    def update(self, score):
        delta = score - self.previous if hasattr(self, 'previous') else 0
        self.ema_delta = self.alpha * delta + (1 - self.alpha) * self.ema_delta
        self.previous = score
        if self.ema_delta < self.min_improvement:
            self.stale_count += 1
        else:
            self.stale_count = 0

    def is_plateau(self):
        return self.stale_count >= self.window
```

**Escape strategies:**
1. Strategy switch (gradient → holistic) — 2 more generations
2. Temperature bump (0.3 → 0.6) — 2 more generations
3. Stop — plateau is real

## 6. Ecosystem: MCP Server

### Design

5 tools exposed via FastMCP (Python MCP SDK), direct Python imports (no subprocess overhead).

| Tool | Purpose | LLM needed? |
|------|---------|:-----------:|
| `schliff_score` | Score a file or inline text | No |
| `schliff_suggest` | Ranked improvement suggestions | No |
| `schliff_verify` | CI-gate: pass/fail against threshold | No |
| `schliff_compare` | Side-by-side comparison of two files | No |
| `schliff_diff` | Explain score changes vs git ref | No |

**No `schliff_fix` or `schliff_evolve` in MCP.** The agent IS the LLM — it uses `score` + `suggest`, edits the file itself, and verifies with `schliff_verify`. This is the idiomatic MCP pattern.

## 9. Competitive Landscape

### Direct Competitors

| Tool | Focus | Stars | Schliff Advantage |
|------|-------|------:|-------------------|
| cclint | Claude Code schema validation | 13 | Semantic quality (not just schema), multi-format, anti-gaming |
| AgentLinter | 8-dim scoring, web-first | ~50 | Eval-suite integration, auto-evolution, drift/sync/track, zero deps |
| agent-friend | MCP schema quality | ~20 | Different scope (instructions vs schemas), complementary |
| cursor-doctor | 100+ checks for .cursorrules | ~100 | Multi-format, autonomous improvement loop, eval-suite |
| Packmind | 17 evaluators, enterprise | — | Transparent scoring, open source, deterministic, no vendor lock-in |
| LobeHub validator | LLM-based, 9 criteria | — | Deterministic (same score every run), CI-tauglich, zero cost |

### Biggest Threat (12 months)

**Anthropic builds it in.** Probability: ~60%. Defense: set the standard before they do. If enough READMEs carry a Schliff badge, Anthropic integrates rather than rebuilds.

### Features to Adopt from Competitors

| Feature | Source | Priority | Effort |
|---------|--------|----------|--------|
| Social sharing (Twitter scorecard) | AgentLinter | HIGH | Small |
| Percentile ranking | AgentLinter | MEDIUM | Medium |
| VS Code extension | cursor-doctor | HIGH | Large (defer) |
| Stack detection | cursor-doctor | MEDIUM | Medium |
| Content marketing with data | cursor-doctor | HIGH | Medium |

### Schliff's Moats

1. **Eval-suite integration** — no competitor connects static analysis with assertion-based testing
2. **Anti-gaming** (6 patterns) — no competitor has this
3. **Autonomous improvement loop** — closed-loop, not just diagnosis
4. **Drift + sync + track** — quality over time
5. **Zero dependencies** — supply-chain-safe, air-gapped-capable
6. **9 dimensions with transparent weights** — documented, auditable, reproducible
